================================================================================
THE PROBLEM WE'RE SOLVING
================================================================================

Companies use OpenAI/Anthropic APIs. They have no idea:
- How much they're spending (until the bill arrives)
- Which projects/teams are burning money
- What prompts are being called repeatedly (cache opportunity)
- When they're about to blow their budget

Our solution: A TRANSPARENT PROXY that sits between them and the LLM providers.

Every API call goes through us. We:
1. Track usage (tokens, cost, latency)
2. Write it to a durable log (WAL)
3. Forward the request like nothing happened

Phase 2 (already built):
- Drain the log into Postgres via a separate binary (waldrain)
- Tamper-evident hash chain (EU AI Act compliance)
- Prompt normalization for dedup
- Chain restart markers for crash recovery

Later (Phase 3+):
- Enforce budget caps (return 429 before they overspend)
- Cache identical requests in Redis
- Send daily Slack reports with spending breakdown

```go
func (w *Writer) syncLoop() {
    ticker := time.NewTicker(100 * time.Millisecond)
    for {
        select {
        case <-ticker.C:  // ← FIRES NOW
            w.mu.Lock()
            if w.file != nil && w.pending > 0 {
                w.file.Sync()  // ← FSYNC TO DISK
                // Takes ~0.5ms on SSD
                w.pending = 0
            }
            w.mu.Unlock()
        }
    }
}
```

What Sync() does:
  - Flushes file buffer to kernel
  - Tells kernel: "Write this to physical disk NOW"
  - Waits for disk controller to confirm write
  - Returns when data is durable (survives power loss)

After this completes:
  - wal-2026-05-21.jsonl is on physical disk
  - Even if server crashes, data survives ✓
  - Reconciliation (Phase 2) will read this file and insert to Postgres

WHAT THE DRAIN WORKER DOES (PHASE 2 — BUILT)
------------------------------------------------------------------------

The WAL drain worker is a separate binary (cmd/waldrain/main.go) that:

1. Watches WAL_DIR for wal-*.jsonl files
2. Reads from wal-offset.json position (file + line number)
3. Batches 100 records per Postgres transaction
4. Verifies hash chain on every batch:
   - batch[0].prev_hash must match last known hash
   - Internal chain links verified within batch
   - Each record's hash recomputed and checked
   - HALTS on any mismatch (tamper detection)
5. Loops drainFile until EOF per tick (no throughput ceiling)
6. Executes Postgres transaction:
   ```sql
   BEGIN;

   INSERT INTO wal_records (
     time, project, provider, model, prompt_hash,
     input_tokens, output_tokens, total_tokens,
     cost, latency_ms, status_code, cache_hit,
     stream, loop_signal, prev_hash, record_hash
   ) VALUES (...)
   ON CONFLICT (record_hash) DO NOTHING;  -- idempotent

   -- Only upsert prompts if the wal_record was new (not a duplicate):
   INSERT INTO prompts (...) VALUES (...)
   ON CONFLICT (project_id, prompt_hash) DO UPDATE
     SET total_calls = prompts.total_calls + 1,
         total_cost = prompts.total_cost + EXCLUDED.total_cost,
         last_seen = NOW();

   COMMIT;
   ```

7. Update offset file: wal-offset.json
   ```json
   {"file":"wal-2026-05-21.jsonl","line":257,"last_hash":"a3f9..."}
   ```
   (Tracks file name, physical line number, and last chain hash)

8. Polls every 5 seconds

Chain restart markers (ChainRestart=true) are recognized as expected
boundaries from crash recovery — not tampering.

Postgres now has queryable data:
  - "Show me all requests for acme-chatbot today"
  - "What's the total cost for acme-chatbot this month?"
  - "Which prompts are called most frequently?"
  - "Verify chain integrity across all records"

------------------------------------------------------------------------
THE MAGIC: WHAT MAKES THIS RELIABLE
------------------------------------------------------------------------

❶ WAL is LOCAL DISK (not network)
  → Can't fail due to network issues
  → Append is <1ms (faster than Postgres INSERT)

❷ WAL WRITE BEFORE RESPONSE RETURN
  → Write() appends to file before sending response
  → In sync mode: fsync also happens before return (true per-request durability)
  → In batch mode (default): fsync happens within ~100ms via background goroutine
  → If we crash BEFORE sending response, client retries (idempotent)

❸ BATCHED FSYNC
  → 50 records OR 100ms, whichever comes first
  → Amortizes fsync cost over many writes
  → 0.5ms / 50 records = 0.01ms per record

❹ JSONL FORMAT
  → One record per line
  → Partial reads are safe (just skip incomplete line)
  → No need to parse entire file

❺ DAILY ROTATION
  → One file per day
  → Old files don't grow unbounded
  → Easy to archive to S3 after 7 days

❻ IDEMPOTENT DRAIN
  → Drain worker can restart mid-batch (transaction rollback)
  → Offset file tracks progress (file + physical line number)
  → Re-reading same records is safe (ON CONFLICT (record_hash) DO NOTHING)
  → Prompts upsert gated on RowsAffected > 0 (no double-counting)

------------------------------------------------------------------------
WHY THIS IS BETTER THAN "JUST USE POSTGRES"
------------------------------------------------------------------------

NAIVE APPROACH (direct Postgres write):
```
Request → OpenAI → Extract usage → INSERT INTO postgres → Return response
                                    ↑
                                  Takes 5-10ms
                                  Can fail (DB down)
                                  Blocks request
```

At 1000 req/s:
  - Postgres handles 1000 INSERT/s (doable but pushing limits)
  - Each request waits 5-10ms for INSERT
  - If Postgres goes down, ALL requests fail (not just tracking)

OUR APPROACH (WAL + async drain):
```
Request → OpenAI → Extract usage → Append to WAL (0.5ms) → Return response
                                    ↑
                                  Fast (local disk)
                                  Can't fail
                                  Doesn't block

(Background worker, separate process)
WAL → Batch 100 records → Single Postgres transaction → Done
```

At 1000 req/s:
  - WAL handles 1000 appends/s (trivial)
  - Each request waits 0.5ms for WAL write
  - Postgres handles 10 transactions/s (100 records each) ← easy
  - If Postgres goes down, requests still work (WAL buffers)

RESULT: 10x faster, 100x more reliable.

- The proxy container can reach Postgres at hostname "postgres"

- In Docker Compose, traffic never leaves the host machine
- SSL adds latency for zero security benefit locally

func Load(path string) (*Config, error) {
    // 1. Start with hardcoded defaults
    cfg := &Config{
        Server: ServerConfig{Host: "0.0.0.0", Port: 8080},
        Postgres: PostgresConfig{Host: "localhost", Port: 5432, ...},
        ...

    // 2. If YAML exists, load it and merge
    if path != "" {

        yaml.Unmarshal(data, cfg)  // Overwrites defaults
    }

    // 3. Apply env var overrides (highest priority)
    applyEnvOverrides(cfg)

    return cfg
}

PRIORITY ORDER (highest to lowest):

2. YAML file (configs/proxy.yaml)
3. Hardcoded defaults (port: 8080, wal.sync_mode: "batch")

WHY THIS ORDER?
- Env vars are how you override in production (Kubernetes secrets)
- YAML is how you configure locally (checked into git)
- Defaults are fallback (works out of the box)

- Viper is 50KB of code for 20 lines of logic
- Cobra is for CLI apps (flags, subcommands). We're a daemon.

- Stores timestamps in UTC, converts on read

    srv := &http.Server{
        Addr:         ":8080",
        Handler:      r,
        ReadTimeout:  30 * time.Second,   // How long to read request body
        WriteTimeout: 120 * time.Second,  // How long to write response (streaming!)
        IdleTimeout:  120 * time.Second,  // How long to keep connection alive
    }

    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)

    go func() {
        log.Info().Str("addr", ":8080").Msg("starting server")
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal().Err(err).Msg("server error")
        }
    }()

    sig := <-sigCh  // Block here until signal received
    log.Info().Str("signal", sig.String()).Msg("shutting down")

    shutdownCtx, cancel := context.WithTimeout(context.Background(), 15*time.Second)
    defer cancel()

    if err := srv.Shutdown(shutdownCtx); err != nil {
        log.Error().Err(err).Msg("shutdown error")
    }

WHY GRACEFUL SHUTDOWN?
- When Kubernetes kills the pod, it sends SIGTERM
- Without graceful shutdown: In-flight requests get TCP RST (connection reset)
- With graceful shutdown:
  1. Stop accepting new requests
  2. Wait for in-flight requests to finish (up to 15 seconds)
  3. Close connections cleanly
  4. Exit

WHY 15 SECOND TIMEOUT?
- Kubernetes default terminationGracePeriodSeconds is 30s
- We give requests 15s to finish
- Kubernetes gives us 30s total → plenty of buffer

WHY 120s WRITE TIMEOUT?
- Streaming responses can be LONG (GPT-4 reasoning might take 60+ seconds)
- 30s would kill streaming requests mid-response
- 120s is generous but safe

      |       fsync() to disk (durability guarantee)
      |
      |-- 8. Return response to client
      |       Streaming: Forward SSE events (but HIDE usage-only chunk)
      |       Non-streaming: Forward JSON response as-is
      |
      v

var Registry = map[string]*Provider{}

func Register(p *Provider) {
    Registry[p.PathPrefix] = p
}

func Lookup(path string) *Provider {
    for prefix, p := range Registry {
        if len(path) >= len(prefix) && path[:len(prefix)] == prefix {
            return p
        }
    }
    return nil
}

HOW IT WORKS:
1. openai.go calls Register() in init() → adds /openai to registry
2. anthropic.go calls Register() in init() → adds /anthropic to registry
3. Request comes in: /openai/v1/chat/completions
4. Lookup("/openai/v1/chat/completions") → returns openai provider

func extractOpenAIUsage(body []byte) (Usage, error) {
    var resp openaiResponse
    if err := json.Unmarshal(body, &resp); err != nil {
        return Usage{}, err
    }
    if resp.Usage == nil {
        return Usage{Model: resp.Model}, nil  // No usage (error response)

        InputTokens:  resp.Usage.PromptTokens,
        OutputTokens: resp.Usage.CompletionTokens,
        TotalTokens:  resp.Usage.TotalTokens,
        Model:        resp.Model,
    }, nil
}

func extractOpenAIStreamUsage(events []SSEEvent) (Usage, error) {

    for i := len(events) - 1; i >= 0; i-- {
        ev := events[i]
        if ev.Data == "" || ev.Data == "[DONE]" {
            continue

        var chunk openaiStreamChunk
        if err := json.Unmarshal([]byte(ev.Data), &chunk); err != nil {
            continue  // Malformed chunk, skip

            usage.Model = chunk.Model
        }
        if chunk.Usage != nil {
            usage.InputTokens = chunk.Usage.PromptTokens
            usage.OutputTokens = chunk.Usage.CompletionTokens
            usage.TotalTokens = chunk.Usage.TotalTokens
            return usage, nil  // FOUND IT, STOP
        }
    }
    // No usage found (stream_options not injected?), return model only
    return usage, nil
}

    var usage Usage

    for _, ev := range events {
        if ev.Data == "" {
            continue
        }

        switch ev.Event {
        case "message_start":
            // Parse input tokens from initial event
            var msg anthropicStreamMessageStart
            json.Unmarshal([]byte(ev.Data), &msg)
            if msg.Message != nil {
                usage.Model = msg.Message.Model
                if msg.Message.Usage != nil {
                    usage.InputTokens = msg.Message.Usage.InputTokens
                }
            }

        case "message_delta":
            // Parse output tokens from final event
            var delta anthropicStreamMessageDelta
            json.Unmarshal([]byte(ev.Data), &delta)
            if delta.Usage != nil {
                usage.OutputTokens = delta.Usage.OutputTokens
            }
        }
    }

    usage.TotalTokens = usage.InputTokens + usage.OutputTokens
    return usage, nil
}

func ResolveProject(r *http.Request) string {
    // Priority 1: X-Project header
    if project := r.Header.Get("X-Project"); project != "" {
        return project
    }

    // Priority 2: SHA256(Authorization header)
    if auth := r.Header.Get("Authorization"); auth != "" {
        hash := sha256.Sum256([]byte(auth))
        return fmt.Sprintf("key:%x", hash[:16])  // 16 bytes = 32 hex chars
    }

    // Priority 3: Unknown
    return "unknown"
}

WHY THIS PRIORITY?
1. X-Project is explicit (customer tells us which project)
2. Authorization hash is implicit (stable per API key)
3. "unknown" is fallback (should be rare)

WHY SHA256(Authorization)?
- Same API key → same project identifier (deterministic)
- Different API keys → different project identifiers (unique)
- Hash is one-way (can't reverse to get the API key)

EXAMPLE:
    Authorization: Bearer sk-test-abc123
    SHA256("Bearer sk-test-abc123") = a1b2c3d4e5f6g7h8...
    project = "key:a1b2c3d4e5f6g7h8"

WHY ONLY 16 BYTES?
- 16 bytes = 128 bits of entropy (2^128 combinations)
- Collision probability: ~0% for millions of API keys
- Full 32 bytes (256 bits) is overkill (wastes DB space)

type Writer struct {
    dir               string
    syncMode          string     // "batch" or "sync"
    mu                sync.Mutex
    file              *os.File
    fileDate          string
    pending           int
    done              chan struct{}
    closeOnce         sync.Once
    lastHash          string     // Phase 2: hash of last written record
    chainDirty        bool       // Phase 2: true if chain head needs saving
    needsChainRestart bool       // Phase 2: true if crash gap detected on init
}

WHY sync.Mutex?
- Multiple goroutines can call Write() concurrently
- Mutex ensures only one goroutine appends at a time
- Without mutex: Writes would interleave → corrupted JSONL

WHY closeOnce?
- Close() can be called multiple times (defer + explicit call)
- Without sync.Once: Second close() would panic (closing closed channel)
- With sync.Once: Second close() is a no-op

func NewWriter(dir string, syncMode string) (*Writer, error) {
    os.MkdirAll(dir, 0755)

    // Phase 2: Load chain head, detect crash gaps
    savedHash, _ := loadChainHead(dir)
    lastHash := savedHash
    needsRestart := false

    diskHash, _ := LastRecordHashOnDisk(dir)
    if diskHash != "" && diskHash != savedHash {
        // Crash gap: records on disk beyond saved chain head.
        // Will emit a chain restart marker on first Write().
        lastHash = diskHash
        needsRestart = true
    }

    if syncMode != "sync" {
        syncMode = "batch"
    }

    w := &Writer{
        dir:               dir,
        syncMode:          syncMode,
        done:              make(chan struct{}),
        lastHash:          lastHash,
        needsChainRestart: needsRestart,
    }
    go w.syncLoop()  // Background goroutine for batched fsync
    return w, nil
}

WHY CRASH GAP DETECTION?
- Chain head is only saved on fsync (not every write)
- After a hard crash, records may have made it to disk but chain head is stale
- Without detection: Next record's prev_hash won't match → drain wedges forever
- With detection: Writer emits a chain restart marker → drain continues

WHY BACKGROUND GOROUTINE?
- Fsync every 100ms (even if < 50 records)
- Ensures records hit disk within 100ms worst-case
- Trade-off: 100ms latency vs. 1ms (fsync every write)
- In sync mode: goroutine still runs but does nothing (sync happens in Write())

func (w *Writer) Write(rec Record) error {
    rec.Time = time.Now().UTC()

    w.mu.Lock()
    defer w.mu.Unlock()

    // Phase 2: Emit chain restart marker if crash gap was detected on init.
    if w.needsChainRestart {
        w.writeChainRestartMarkerLocked()  // Bridges the gap
        w.needsChainRestart = false
    }

    // Phase 2: Apply hash chaining (sets prev_hash and record_hash)
    Chain(&rec, w.lastHash)

    line, _ := json.Marshal(rec)
    line = append(line, '\n')

    w.ensureFile()
    w.file.Write(line)

    w.lastHash = rec.RecordHash
    w.chainDirty = true
    w.pending++

    if w.syncMode == "sync" {
        // SYNC MODE: fsync on every write (per-request durability)
        w.file.Sync()
        w.pending = 0
        saveChainHead(w.dir, w.lastHash)
        w.chainDirty = false
    } else {
        // BATCH MODE: fsync every 50 records
        if w.pending >= 50 {
            w.file.Sync()
            w.pending = 0
            saveChainHead(w.dir, w.lastHash)
            w.chainDirty = false
        }
    }

    return nil
}

THE FSYNC GUARANTEE (CONFIGURABLE):
- Without fsync: Data is in kernel buffer, not on disk
- If kernel crashes, data is lost (rare but happens)
- With fsync: Data is on physical disk, durable
- Cost: ~0.5ms for HDD, ~0.05ms for SSD

TWO MODES (configurable via wal.sync_mode or WITNESS_WAL_SYNC_MODE):

  BATCH MODE (default):
  - Fsync every 50 records OR every 100ms via background goroutine
  - Write() returns after OS buffer write, NOT after fsync
  - Up to ~100ms of records can be lost on hard crash
  - 50 writes * 0.05ms fsync = 0.001ms per write (negligible overhead)
  - Best for: high throughput, where ~100ms loss window is acceptable

  SYNC MODE (wal.sync_mode: "sync"):
  - Fsync on EVERY Write() call before returning
  - True per-request durability — zero data loss on crash
  - Higher latency per write (~0.5ms on SSD)
  - Best for: compliance, audit logs, when you can't lose a single record

WHY DOES FSYNC TIMING MATTER?
- If we return response WITHOUT fsyncing:
  - Client thinks request succeeded
  - Kernel crashes before fsync
  - WAL doesn't have the record
  - Reconciliation: Customer was charged by OpenAI, we have no record
  - Customer: "Where's my money going?!"
  - You: "Uhh... we lost your data"
  - Customer: Fires you, switches to competitor

- This is why SYNC MODE exists:
  - Kernel crashes after fsync → WAL has the record (durable)
  - Reconciliation: WAL matches OpenAI bill ✅
  - Customer: Happy, renews contract

- BATCH MODE accepts ~100ms of risk for performance:
  - In practice, kernel crashes are rare
  - 100ms window means at most a few records lost
  - For most use cases, this trade-off is fine
  - For EU AI Act audit logs, use sync mode

func (w *Writer) ensureFile() error {
    today := time.Now().UTC().Format("2006-01-02")  // "2025-01-20"
    if w.file != nil && w.fileDate == today {
        return nil  // File is still valid
    }

    // Date changed (midnight UTC) or first boot
    if w.file != nil {
        w.file.Sync()   // Flush old file
        w.file.Close()
        w.pending = 0
    }

    path := filepath.Join(w.dir, fmt.Sprintf("wal-%s.jsonl", today))
    f, err := os.OpenFile(path, os.O_CREATE|os.O_APPEND|os.O_WRONLY, 0644)
    if err != nil {
        return err
    }

    w.file = f
    w.fileDate = today
    return nil
}

FILE ROTATION:
- One file per day: wal-2025-01-20.jsonl, wal-2025-01-21.jsonl, ...
- At midnight UTC, new file is created
- Old file is closed (but kept on disk)
- Phase 2: Drain worker processes old files
- Phase 5: Archive old files to S3 after 7 days

WHY DAILY ROTATION?
- Files don't grow unbounded (max 1 day of data per file)
- Drain worker can process completed days in parallel
- Easy to archive: "Move all wal-*.jsonl older than 7 days to S3"

func (w *Writer) syncLoop() {
    ticker := time.NewTicker(100 * time.Millisecond)
    defer ticker.Stop()
    for {
        select {
        case <-w.done:
            return  // Close() was called, exit
        case <-ticker.C:
            w.mu.Lock()
            if w.file != nil && w.pending > 0 {
                w.file.Sync()
                w.pending = 0
            }
            w.mu.Unlock()
        }
    }
}

WHAT THIS DOES:
- Every 100ms, check if there are pending writes
- If yes, fsync to disk
- If no, do nothing

WHY 100ms?
- 10 fsyncs per second = 0.5ms total overhead per second
- Max latency: 100ms (if record arrives right after tick)
- Average latency: 50ms (assuming uniform distribution)

func (h *Handler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    provider := providers.Lookup(r.URL.Path)
    if provider == nil {
        http.Error(w, `{"error":"unknown provider"}`, 404)
        return
    }

    // Read request body
    body, err := io.ReadAll(r.Body)
    if err != nil {
        http.Error(w, `{"error":"failed to read request body"}`, 400)
        return
    }
    r.Body.Close()

    // Resolve attribution
    project := ResolveProject(r)

    // Check if streaming
    isStream := provider.IsStreamRequest != nil && provider.IsStreamRequest(body)

    if isStream {
        h.handleStream(w, r, provider, body, project)
        return
    }

    h.handleNonStream(w, r, provider, body, project)
}

PATH STRIPPING:
    Client sends:     /openai/v1/chat/completions
    We strip:         /openai
    Upstream sees:    /v1/chat/completions
    Upstream URL:     https://api.openai.com/v1/chat/completions

func (h *Handler) handleNonStream(...) {
    start := time.Now()

    // Build upstream URL
    upstreamPath := strings.TrimPrefix(r.URL.Path, provider.PathPrefix)
    upstreamURL := provider.Target.String() + upstreamPath
    if r.URL.RawQuery != "" {
        upstreamURL += "?" + r.URL.RawQuery
    }

    // Create upstream request
    upReq, _ := http.NewRequestWithContext(r.Context(), r.Method, upstreamURL, bytes.NewReader(body))

    // Copy headers (Authorization, Content-Type, etc.)
    copyHeaders(r.Header, upReq.Header)
    upReq.Header.Set("Content-Length", fmt.Sprintf("%d", len(body)))

    // Forward to upstream
    resp, err := h.Transport.RoundTrip(upReq)
    if err != nil {
        http.Error(w, `{"error":"upstream request failed"}`, 502)
        return
    }
    defer resp.Body.Close()

    // Read response
    respBody, _ := io.ReadAll(resp.Body)
    latency := time.Since(start)

    // Extract usage
    var usage providers.Usage
    if resp.StatusCode >= 200 && resp.StatusCode < 300 {
        usage, _ = provider.ExtractUsage(respBody)
    }

    // Compute cost
    cost := providers.ComputeCost(usage.Model, usage.InputTokens, usage.OutputTokens)

    // Hash prompt
    promptHash := hashPrompt(body)

    // Normalize prompt + hash (Phase 2: strips timestamps, UUIDs, etc.)
    promptHash := hashPrompt(body)
    // hashPrompt calls attribution.HashNormalizedPrompt(string(body))

    // ⚠️ CRITICAL: Write WAL BEFORE returning response ⚠️
    // Hash chain (prev_hash, record_hash) is applied inside Write()
    h.WAL.Write(wal.Record{
        Project:      project,
        Provider:     provider.Name,
        Model:        usage.Model,
        PromptHash:   promptHash,
        InputTokens:  usage.InputTokens,
        OutputTokens: usage.OutputTokens,
        TotalTokens:  usage.TotalTokens,
        Cost:         cost,
        LatencyMs:    latency.Milliseconds(),
        StatusCode:   resp.StatusCode,
        CacheHit:     false,
        Stream:       false,
    })

    // Write response to client
    copyHeaders(resp.Header, w.Header())
    w.WriteHeader(resp.StatusCode)
    w.Write(respBody)
}

THE ORDER MATTERS:
    1. Forward to OpenAI
    2. Get response
    3. Extract usage
    4. Compute cost
    5. Normalize prompt + hash (Phase 2)
    6. Write to WAL (hash chained, then fsync in sync mode)
    7. Return response to client

In SYNC MODE: Step 6 blocks until fsync → true per-request durability.
In BATCH MODE: Step 6 writes to OS buffer → fsync happens within ~100ms.

If we did 7 before 6, and the server crashed at step 7, we'd lose the record.

func (h *Handler) handleStream(...) {
    // Prepare body (inject stream_options for OpenAI)
    if provider.PrepareStreamBody != nil {
        body, _ = provider.PrepareStreamBody(body)
    }

    // Forward to upstream (same as non-streaming)
    upstreamURL := ...
    upReq, _ := http.NewRequestWithContext(...)
    resp, _ := h.Transport.RoundTrip(upReq)

    // If upstream returned non-SSE (error), pass through
    contentType := resp.Header.Get("Content-Type")
    if !strings.Contains(contentType, "text/event-stream") {
        respBody, _ := io.ReadAll(resp.Body)
        w.WriteHeader(resp.StatusCode)
        w.Write(respBody)
        return
    }

    // Set up streaming response to client
    flusher := w.(http.Flusher)
    w.Header().Set("Content-Type", "text/event-stream")
    w.WriteHeader(resp.StatusCode)

    // Accumulate events while streaming to client
    var events []providers.SSEEvent
    scanner := bufio.NewScanner(resp.Body)
    scanner.Buffer(make([]byte, 0, 256*1024), 1024*1024)  // 256KB initial, 1MB max

    var currentEvent providers.SSEEvent

    for scanner.Scan() {
        line := scanner.Text()

        if line == "" {
            // Empty line = end of event
            if currentEvent.Data != "" || currentEvent.Event != "" {
                events = append(events, currentEvent)

                // Check if this is usage-only chunk (OpenAI)
                isUsageChunk := false
                if provider.Name == "openai" && currentEvent.Data != "[DONE]" {
                    isUsageChunk = isOpenAIUsageOnlyChunk(currentEvent.Data)
                }

                if !isUsageChunk {
                    // Forward event to client
                    writeSSEEvent(w, currentEvent)
                    flusher.Flush()
                }

                currentEvent = providers.SSEEvent{}
            }
            continue
        }

        if strings.HasPrefix(line, "event:") {
            currentEvent.Event = strings.TrimSpace(strings.TrimPrefix(line, "event:"))
        } else if strings.HasPrefix(line, "data:") {
            data := strings.TrimPrefix(line, "data:")
            data = strings.TrimPrefix(data, " ")
            if currentEvent.Data != "" {
                currentEvent.Data += "\n" + data  // Multi-line data
            } else {
                currentEvent.Data = data
            }
        }
    }

    latency := time.Since(start)

    // Extract usage from accumulated events
    usage, _ := provider.ExtractStreamUsage(events)
    cost := providers.ComputeCost(usage.Model, usage.InputTokens, usage.OutputTokens)

    // ⚠️ Write WAL BEFORE connection closes ⚠️
    h.WAL.Write(wal.Record{
        ...
        Stream: true,
    })
}

WHY ACCUMULATE EVENTS?
- We need ALL events to extract usage (it's at the end)
- We can't wait to forward (client wants immediate streaming)
- Solution: Accumulate in memory, forward in real-time

MEMORY USAGE:
- Typical streaming response: 50-100 events
- Each event: ~200 bytes
- Total: 10-20KB per request
- Max concurrent streams: 1000
- Peak memory: 20MB (negligible)

WHY FILTER USAGE-ONLY CHUNK?
- OpenAI SDK doesn't expect this chunk (we injected stream_options)
- If we forward it: SDK sees unknown chunk → logs warning → customer confused
- If we filter it: SDK works normally, we capture usage silently

USAGE-ONLY CHUNK:
    data: {"id":"c","model":"gpt-4o-mini","choices":[],"usage":{"prompt_tokens":10,"completion_tokens":2,"total_tokens":12}}

    choices=[] (no delta) + usage present = USAGE-ONLY

func isOpenAIUsageOnlyChunk(data string) bool {
    var chunk struct {
        Usage   *json.RawMessage
        Choices []struct {
            Delta struct {
                Content *string
            }
        }
    }
    json.Unmarshal([]byte(data), &chunk)

    if chunk.Usage == nil {
        return false  // No usage
    }
    if len(chunk.Choices) == 0 {
        return true  // Usage + no choices = usage-only
    }
    for _, c := range chunk.Choices {
        if c.Delta.Content != nil && *c.Delta.Content != "" {
            return false  // Has content
        }
    }
    return true  // Usage + empty/null content = usage-only
}

DESIGN DECISIONS WE'D CHANGE (IF STARTING OVER):
1. Use ClickHouse instead of Postgres?
   → NO. Postgres is simpler, sufficient for 10M req/day
   → ClickHouse adds operational complexity for zero benefit early on

2. Use Kafka instead of WAL?
   → NO. Kafka is 3 containers + Zookeeper + partition management
   → WAL is 1 directory on disk + 200 lines of code
   → Kafka makes sense at 100M+ req/day (not now)

3. Use gRPC instead of HTTP?
   → NO. Every LLM provider uses HTTP/SSE
   → gRPC would require protocol translation (adds latency)
   → HTTP is the right choice

4. Use TypeScript/Node.js instead of Go?
   → NO. Node.js is slower (V8 JIT vs. compiled)
   → Goroutines are lighter than threads (10K concurrent streams is easy)
   → Go binary is self-contained (Node needs runtime + node_modules)

Full request flow (240µs total):

1. Receive request:          10µs  (parse HTTP headers)
2. Resolve project:          0.5µs (SHA256 hash)
3. Check if streaming:       2µs   (JSON decode, check "stream" field)

8. WAL write:                14µs  (hash chain + JSON encode + append)

TOTAL: 220.5µs ≈ 240µs (rounding + syscall overhead)

POST-PHASE 2 FIXES (AFTER INITIAL BUILD)
========================================================================

Three bugs found and fixed after Phase 2 was declared complete:

1. STREAMING MEMORY ACCUMULATION
   - Bug: handleStream stored EVERY SSE event in memory. Long agent streams
     (thousands of content deltas) risked OOM under concurrency.
   - Fix: Changed `events` to `usageEvents`. Only keeps usage-relevant events
     (first event + usage-only chunks for OpenAI; message_start + message_delta
     for Anthropic). Content deltas are forwarded to the client but never stored.
   - File: internal/proxy/streaming.go (isUsageRelevantEvent function)

2. USAGE CHUNK SUPPRESSION BROKE CLIENTS
   - Bug: Proxy always suppressed OpenAI usage-only chunks. But if the CLIENT
     had already set stream_options.include_usage=true, they needed that chunk
     for their own accounting. We were eating it.
   - Fix: Added clientHasIncludeUsage() check before PrepareStreamBody. Tracks
     `weInjectedUsage` bool. Suppression only happens when WE injected it.
   - File: internal/proxy/streaming.go (clientHasIncludeUsage function)
   - Test: TestHandler_StreamOpenAI_ClientRequestedUsage

3. WAL WRITE FAILURES WERE SILENT
   - Bug: WAL write failure just logged and continued. No metric. No way to
     alert on it. Silent data loss.
   - Fix: Added llmproxy_wal_write_failures_total Prometheus counter. Incremented
     on every WAL write failure in both streaming and non-streaming paths.
   - File: internal/proxy/handler.go (walWriteFailures counter)

Three repo hygiene issues fixed:

4. COMMITTED BINARIES (41MB)
   - proxy.exe and waldrain.exe were committed to git. Removed from tracking,
     added /waldrain to .gitignore (*.exe was already there).

5. DEAD REQUESTS TABLE
   - The `requests` table in schema.sql was superseded by `wal_records`.
     Two tables for the same data. Dropped requests and its 3 indexes.
     Schema now has 6 tables, down from 7.

6. PHONE REGEX OVER-NORMALIZATION
   - Bug: Phone regex (\d{3}...\d{4}) matched any 10-digit run in JSON.
     Token IDs like "9876543210" were being replaced with <PHONE>, clustering
     unrelated prompts under the same hash.
   - Fix: Changed separators from optional ([-.\s]?) to required ([-.\s]).
     Now requires dashes, dots, or spaces between digit groups.
     "555-123-4567" still matches. "5551234567" does not.
   - File: internal/attribution/normalizer.go (phoneRegex)
   - Test: TestNormalizePhone (shouldMatch + shouldNotMatch assertions)

MVP SCOPE CHANGE:
   - Removed Gemini and Cohere providers from Phase 5. MVP ships with
     OpenAI + Anthropic only. New providers can be added post-launch
